# Migrating to FlowCraft `sdk/v0.3.0`

> Status: planning. This document tracks the breaking changes scheduled
> for `sdk/v0.3.0` so users can prepare in advance and contributors do
> not introduce new API in soon-to-move locations.

## Summary

`sdk/v0.3.0` is the cleanup release that finishes the `sdk` ↔ `sdkx`
layering rule:

> **`sdk` defines interfaces and primitives. `sdkx` ships concrete
> adapters that integrate with external systems or external protocol
> specs.**

The rule is already followed by `sdk/llm` (interface) → `sdkx/llm/{qwen,
openai, deepseek, minimax, bytedance}` (concrete provider adapters).
The same shape will be applied to LLM-callable tools, which today
straddle the boundary inconsistently.

`sdk/v0.3.0` will also delete the long-deprecated `sdk/workflow`
package (announcement: `CLAUDE.md`, original deprecation: v0.1.x).

---

## Breaking changes

### 1. LLM tool implementations move to `sdkx/tool/<service>`

The following tool implementations move out of the service packages
they currently live next to. The function/struct shapes are preserved
across the move; only the import path changes.

| Today (`sdk` v0.2.x)                              | After (`sdk/v0.3.0` + `sdkx/v0.3.0`)             |
| ------------------------------------------------- | ------------------------------------------------ |
| `sdk/knowledge.NewSearchServiceTool`              | `sdkx/tool/knowledge.NewSearchServiceTool`       |
| `sdk/knowledge.NewPutServiceTool`                 | `sdkx/tool/knowledge.NewPutServiceTool`          |
| `sdk/kanban.SubmitTool`                           | `sdkx/tool/kanban.SubmitTool`                    |
| `sdk/kanban.TaskContextTool`                      | `sdkx/tool/kanban.TaskContextTool`               |
| `sdk/kanban.WithKanban` / `sdk/kanban.KanbanFrom` | `sdkx/tool/kanban.WithKanban` / `KanbanFrom`     |
| `sdk/history.ToolDeps`                            | `sdkx/tool/history.ToolDeps`                     |
| `sdk/history.RegisterTools`                       | `sdkx/tool/history.RegisterTools`                |

**What stays in `sdk`:**

- `sdk/agent/handoff.go` — Handoff Tool + Decider are an agent runtime
  orchestration pair, not an external service adapter. Their
  detection logic and tool naming convention live in the same package
  by design.
- The `sdk/tool` interface package itself (`Tool`, `Registry`,
  `FuncTool`, `DefineSchema`, …) — this is the contract layer.

**What is added new in `sdkx`:**

- `sdkx/tool/memory` — Anthropic Memory Tool spec-compatible wrapper
  over `sdk/workspace.Workspace` (currently being designed; will land
  in `sdkx/v0.2.x` ahead of the `sdk` move).
- `sdkx/tool/mcp` — MCP client primitive, exposes external MCP
  servers as `tool.Tool` instances.

#### Migration

Mechanical import-path swap:

```diff
-import "github.com/GizClaw/flowcraft/sdk/knowledge"
+import knowledgetool "github.com/GizClaw/flowcraft/sdkx/tool/knowledge"

- search := knowledge.NewSearchServiceTool(svc)
+ search := knowledgetool.NewSearchServiceTool(svc)
```

A migration script (`scripts/migrate-tools-v0.3.go`) will be shipped
that rewrites these imports in-place.

### 2. `sdk/workflow` deletion

`sdk/workflow` has been deprecated since v0.1.x. v0.3.0 removes the
package entirely. The migration path is documented in
[`sdk/workflow/doc.go`](../../sdk/workflow/doc.go) — every concept
maps to a combination of `sdk/agent` + `sdk/engine` + `sdk/graph`.

---

## Why now

The fragmentation was visible since v0.2:

- Provider tools have always lived in `sdkx/llm/*`.
- The `Tool` interface lives in `sdk/tool`.
- But individual `Tool` implementations were left next to the service
  they call (`sdk/knowledge.NewSearchServiceTool`, etc.) for early
  development convenience.

This made `sdk` carry concrete-adapter responsibility (extra
dependencies, faster API churn) that the layering rule explicitly
gives to `sdkx`. v0.3.0 is the cleanup window.

## Backwards-compatibility shim

There is no shim. `sdk` cannot import `sdkx` (that would create an
import cycle since `sdkx` already depends on `sdk`). The migration
must be a hard cut.

To soften the impact:

- All affected symbols are marked `// Deprecated:` from `sdk/v0.2.x`
  onwards, so `go vet` / IDE warnings flag them ahead of the cut.
- The migration script handles the mechanical rewrite for users
  who want a one-liner upgrade.

## Timeline

- **`sdk/v0.2.x`** (current): Deprecation notices added.
  No behavioural change. Users SHOULD start migrating in advance.
- **`sdkx/v0.2.x`** (parallel): New `sdkx/tool/*` packages land
  alongside the existing `sdk` locations so users can switch on
  their own schedule.
- **`sdk/v0.3.0`**: Old locations removed; `sdk/workflow` removed.
  Release notes will reference this file.
